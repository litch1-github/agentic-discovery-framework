# Решение: Перепроектирование процесса обработки ипотеки

## ТЕКУЩИЙ ПРОЦЕСС (AS-IS)
1. Клиент подает заявку через сайт/отделение (1 день)
2. Оператор проверяет полноту пакета документов (1 день)
3. Андеррайтер запрашивает данные из БКИ, ФНС (1-2 дня)
4. Скоринг-модель оценивает риск (1 день)
5. Андеррайтер изучает кейс вручную (2-3 дня)
6. Руководитель согласовывает решение (1 день)
ИТОГО: 7-10 дней (последовательный процесс)

## АДАПТАЦИЯ (если автоматизируем старый процесс)
Вставляем AI-ассистента на отдельных шагах:
- AI помогает оператору проверять документы (OCR) → экономия 0.5 дня
- AI готовит резюме кейса для андеррайтера → экономия 1 день
- AI автоматически запрашивает данные из БКИ → экономия 0.5 дня

Результат: 7-10 дней → 5-6 дней (ускорение в 1.5 раза)
Прирост: ~25-30%
Организационные изменения: минимальные

## ПЕРЕПРОЕКТИРОВАНИЕ (если меняем процесс вокруг возможностей агентов)
Полностью перестраиваем процесс:
1. Клиент дает согласие на анализ цифрового следа (15 минут)
2. AI-агент параллельно:
   - Запрашивает данные из БКИ, ФНС, ЕГРЮЛ через API (5 минут)
   - Запускает скоринг-модель (10 минут)
   - Анализирует документы через компьютерное зрение (10 минут)
3. AI-агент принимает решение:
   - 80% заявок → автоматическое одобрение/отказ (30 минут)
   - 20% заявок → эскалация на андеррайтера с готовым резюме (2-4 часа)

Результат: 7-10 дней → 4 часа для 80% клиентов (ускорение в 48 раз)
Прирост: ~200-300%
Организационные изменения: значительные (перестройка процесса, обучение команды)

## РЕШЕНИЕ
Выбираем ПЕРЕПРОЕКТИРОВАНИЕ.

## ОБОСНОВАНИЕ
1. Конкуренты уже ушли в отрыв (1-2 дня). Адаптация даст нам 5-6 дней — мы все 
   равно проиграем.
2. Текущий процесс исторически сложился вокруг бумажного документооборота и 
   ручных проверок. Если мы просто "ускорим" его, мы останемся в старой парадигме.
3. AI-агенты позволяют параллельные проверки и автоматическую эскалацию — это 
   дает нелинейный прирост.
4. Риск-блок готов к перепроектированию: у нас есть зрелая скоринг-модель и 
   опыт автоматизации.
5. Fallback-критерий защитит нас: если гипотеза не сработает, мы откатимся к 
   ручному процессу.

## РИСКИ ПЕРЕПРОЕКТИРОВАНИЯ
- Высокие организационные изменения (обучение команды, изменение KPI)
- Необходимость интеграции с внешними API (БКИ, ФНС)
- Регуляторные требования (152-ФЗ, ЦБ)
- Доверие клиентов к автоматическому решению

## МИТИГАЦИИ
- Пилот на сегменте "Новостройки" (низкий риск, стандартные кейсы)
- Поэтапный rollout: 10% → 30% → 80% заявок
- Еженедельный аудит одобренных заявок риск-блоком
- Возможность апелляции для клиентов
